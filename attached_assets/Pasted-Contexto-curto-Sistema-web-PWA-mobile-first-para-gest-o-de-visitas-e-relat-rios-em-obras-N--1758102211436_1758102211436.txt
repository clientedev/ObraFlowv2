Contexto curto:
- Sistema web/PWA mobile-first para gestão de visitas e relatórios em obras. 
- NÃO altere o comportamento atual: a primeira tela após login continua sendo o DASHBOARD.
- Objetivo: implementar, com profissionalismo e clareza, as 3 melhorias descritas abaixo. Peço código claro e minimalista, focado em responsividade/mobile, e instruções de teste.

Requisitos (implemente exatamente o que está descrito):

1) Botão direto para criação de relatório (comportamento correto)
- Ao ABRIR a tela de VISUALIZAÇÃO de um projeto (ou seja, quando o usuário clicar para ver os detalhes do projeto a partir do dashboard), deve existir, no INÍCIO/topo da visualização do projeto, um botão primário e destacado: "Criar Relatório".
- Não troque a tela inicial do sistema: manter o dashboard como primeira tela após login.
- O botão deve ser claramente visível em telas mobile (tamanho, contraste, posição fixa no topo do conteúdo), e ser o primeiro CTA lógico da página de detalhes.
- Ao clicar no botão, o sistema deve redirecionar para a tela de criação de relatório com o projeto identificado (veja item 2).

2) Identificação automática de Projeto e Endereço na criação de relatório
- Ao acionar "Criar Relatório" a partir da VIEW do projeto, a tela de criação deve abrir com:
  - project_id preenchido (oculto) e project_name exibido (não-editável ou editável conforme UX), e
  - endereço do projeto (campo de endereço preenchido com os dados cadastrados).
- Não deve ser necessário procurar o projeto em lista nem clicar na localização para puxar o endereço: preencher automaticamente.
- Comportamento de fallback: se o projeto não tiver endereço cadastrado, mostrar o campo endereço vazio e indicar "Endereço não cadastrado — preencha para continuar".
- A navegação pode usar querystring (ex: /relatorios/novo?project_id=123) ou qualquer método seguro; o resultado final deve ser o mesmo: auto-populado e pronto para edição/assinatura.

3) Configuração de Aprovador Padrão no Admin
- Adicionar no menu de administração uma seção nova: "Configurações do Sistema → Aprovador Padrão".
- A página deve:
  - exibir o aprovador atual (se houver) e permitir buscar/selecionar um usuário ativo para ser o aprovador padrão;
  - salvar a escolha como configuração global (ex: chave `default_report_approver_id` em tabela de configurações/genéricas).
- Regras aplicadas ao criar relatório:
  - Ao criar um novo relatório, se o campo "Aprovador" não for manualmente alterado, o sistema deve preencher automaticamente com o aprovador padrão configurado.
  - O campo "Aprovador" na criação do relatório deve ser editável — o usuário pode sobrescrever o padrão se necessário.
- Permissões: apenas usuários com perfil administrador/master podem acessar e alterar esta configuração.
